So to verify zk circuits we need to generate verification keys. Verification keys are parameters for verifier so that verifier would accept proof only for specific zk circuit. 
`VerifierParams` are also parameters for verifier smart contract, but specific to our circuit architecture. In our architecture we do have recursive proofs. [10/21/2023 11:39] vladbochok
So, a contract can pass non-zero register to the call in case `systemCall` flag is set to true. We use this non-standard ABI for efficiency and to distinguish EVM ABI that users can use. `MsgValueSimulator` is using register for getting (value, to, isSystemCall) parameters to the call that is need to be mimic. In case `to` == `msgValueSimulator` address and `isSystemCall` is set to true then `MsgValueSimulator` reenter itself with registers that caller didn't set in purpose (some not cleaned register that `MsgValueSimulator` used in the execution).  

> due to WILL mess up the registers and WILL use r1-r4 for our standard ABI convention and r5 for the extra who_to_mimic argument or due to the usage of registers in performing operations? 
They are coming due to usage of `isSystemCall` and they are messy due to not specific setting of such register, but rather using register that is left from the execution.

> And what is meant by the return value being WILL mess up the registers and WILL use r1-r4 for our standard ABI convention and r5 for the extra who_to_mimic argument for the mimic_call_byref opcode?
That's being said, register in the frame will be passed to the next frame if `isSystemCall` flag is set.
